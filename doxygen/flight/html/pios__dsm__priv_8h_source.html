00038 /*
00039  * Currently known DSMx (DSM2, DSMJ, DSMX) satellite serial port settings:
00040  *  115200bps serial stream, 8 bits, no parity, 1 stop bit
00041  *  size of each frame: 16 bytes
00042  *  data resolution: 10 or 11 bits
00043  *  number of frames: 1 or 2
00044  *  frame period: 11ms or 22ms
00045  *
00046  * Currently known DSMx frame structure:
00047  *  2 bytes - depend on protocol version:
00048  *    for DSM2/DSMJ:
00049  *      1 byte - lost frame counter (8 bit)
00050  *      1 byte - data format (for master receiver bound with 3 or 5 pulses),
00051  *               or unknown (for slave receiver bound with 4 or 6 pulses,
00052  *               some sources call it also the lost frame counter)
00053  *    for DSMX:
00054  *      1 byte - unknown data (does not look like lost frame counter)
00055  *      1 byte - unknown data, has been seen only 0xB2 so far
00056 
00057  * 14 bytes - up to 7 channels (16 bit word per channel) with encoded channel
00058  *            number, channel value, the "2nd frame in a sequence" flag.
00059  *            Unused channels have FF FF instead of data bytes.
00060  *
00061  * Data format identification:
00062  *   - for DSM2/DSMJ: [0 0 0 R 0 0 N1 N0]
00063  *     where
00064  *        R is data resolution (0 - 10 bits, 1 - 11 bits),
00065  *        N1..N0 is the number of frames required to receive all channel
00066  *        data (01 or 10 are known to the moment, which means 1 or 2 frames).
00067  *        Three values for the transmitter information byte have been seen
00068  *        thus far: 0x01, 0x02, 0x12.
00069  *   - for DSMX this byte contains just 0xB2 or 0xA2 value for any resolution.
00070  *     It is not known at the moment how to find the exact resolution from the
00071  *     DSMX data stream. The frame number (1 or 2) and 10/11 bit resolution were
00072  *     found in different data streams. So it is safer at the moment to ask user
00073  *     explicitly choose the resolution.
00074  *     Also some weird throttle channel (0) behavior was found in some streams
00075  *     from DX8 transmitter (all zeroes). Thus DSMX needs special processing.
00076  *
00077  * Channel data are:
00078  * - for 10 bit: [F 0 C3 C2 C1 C0 D9 D8 D7 D6 D5 D4 D3 D2 D1 D0]
00079  * - for 11 bit: [F C3 C2 C1 C0 D10 D9 D8 D7 D6 D5 D4 D3 D2 D1 D0]
00080  *   where
00081  *      F is normally 0 but set to 1 for the first channel of the 2nd frame,
00082  *      C3 to C0 is the channel number, 4 bit, zero-based, in any order,
00083  *      Dx..D0 - channel data (10 or 11 bits)
00084  *
00085  * DSM2 protocol bug: in some cases in 2-frame format some bytes of the
00086  * frame can contain invalid values from the previous frame. They usually
00087  * are the last 5 bytes and can be equal to FF or other data from last
00088  * frame. There is no explicit workaround currently known.
00089  *
00090  * Binding: the number of pulses within bind window after power up defines
00091  * if this receiver is a master (provides receiver capabilities info to
00092  * the transmitter to choose data format) or slave (does not respond to
00093  * the transmitter which falls back to the old DSM mode in that case).
00094  * Currently known are 3(4) pulses for low resolution (10 bit) DSM2 mode,
00095  * 5(6) pulses for high resolution (11 bit) DSM2 mode, and also 7(8) and
00096  * 9(10) pulses for DSMX modes. Thus only 3, 5, 7 or 9 pulses should be
00097  * used for stand-alone satellite receiver to be bound correctly as the
00098  * master.
00099  */
